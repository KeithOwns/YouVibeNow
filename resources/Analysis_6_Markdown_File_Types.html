The Types of Markdown Files in Your Workspace

An audit of how the same idea — "categorize your project's markdown files" — shows up in two genuinely different, non-contradictory framings across this codebase.

1. The Dashboard Categories (By Agent Utility)

As recorded in project memory and rendered dynamically by the ai_context_tree.html dashboard — your files are aggregated by how LLMs actually ingest them.
  1. CLAUDE.md — instructions auto-loaded by Claude Code (and Claude Cowork's folder instructions).
  2. GEMINI.md — rules auto-loaded by Gemini CLI and Antigravity.
  3. AGENTS.md — universal multi-agent boundary system; Codex CLI and Antigravity read it by default, Gemini CLI only if opted in via context.fileName.
  4. SKILL.md — packaged, reusable workflows loaded on-demand via progressive disclosure (Claude Code and Gemini CLI both support the open Agent Skills standard).
  5. auto-mem_ — Claude Code's per-directory memory, written by Claude across sessions.
  6. gemini-mem_ new — Gemini CLI's per-project memory file, written via its save_memory tool.
  7. other_ — static documentation (READMEs, logs, manuals) that is never auto-loaded, only read on demand.

2. The Standard Project-Level Files (Vibe Coding Starter Kit)

As documented in the YouVibeNow course curriculum — the standard set of files every codebase or workspace folder should contain to maximize stateful prompting performance.
  1. AGENTS.md — anchors the agent's identity, role, and absolute safety constraints (e.g., backup requirements).
  2. CONTEXT.md — outlines the project background, tech stack, directory layouts, and "what good looks like" definitions.
  3. TASKS.md — keeps track of the active sprint with structured Markdown checklists.
  4. MEMORY.md — acts as the dynamic brain, logging learned preferences, self-corrections, and design pivots.
  5. SKILL.md (inside skills/) — packages repetitive, complex multi-step execution tasks into single-command automation macros.
  6. README.md — serves as the human-facing front-door introducing the repository.

Reconciling the Two Framings

Added during review — these are genuinely two different axes, not a contradiction.
Framing 1

What a tool auto-loads, by filename — the dashboard's actual type/scope taxonomy.

Framing 2

What a new project should scaffold — a content-authoring checklist for this curriculum's "starter kit," largely orthogonal to auto-load behavior.

CONTEXT.md/TASKS.md/README.md are never auto-loaded by any tool — they're read only because an AGENTS.md/CLAUDE.md instruction or the user points the agent at them.

Only Framing 1's filenames are confirmed auto-loaded by real tools: CLAUDE.md, GEMINI.md, AGENTS.md (opt-in for Gemini CLI), SKILL.md. README.md is never auto-loaded by any AI tool — though GitHub itself auto-renders it as a folder's landing page, a separate platform convention.
