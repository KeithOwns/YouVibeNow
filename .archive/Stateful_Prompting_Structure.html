Stateful Prompting Root File Structure

The optimal file architecture for an LLM-assisted project, replacing "monolithic prompts" with environmental context.

📄 README.md

Human-AI Bridge
  • Purpose: The highest-level elevator pitch of the repository.
  • Contents: Project summary, goals, and setup instructions. It's the first file an LLM (and a human) reads to orient itself.

📄 AGENTS.md

Prompt Element: Identity & Global Constraints
  • Purpose: Establishes who the AI is and sets strict, global boundaries.
  • Contents: Persona, primary role, and non-negotiable rules (e.g., "always write plain language," "always backup files").
📚 Authority: Supported by Andrej Karpathy's "State of GPT" (emphasizing "System Prompts" to lock in persona) and the Anthropic Prompt Engineering Guide (advocating for "Role-Playing" to drastically boost output quality).

📄 CONTEXT.md

Prompt Element: Context
  • Purpose: Provides the deep background so you don't have to re-explain the project in every chat.
  • Contents: Project summary, target audience, tech stack, and definitions of "what good looks like."
📚 Authority: Aligns with the OpenAI Prompt Engineering Guide core tactic to "Provide reference text."

📄 MEMORY.md

Prompt Element: Dynamic Constraints
  • Purpose: Acts as the AI's long-term brain across different chat sessions.
  • Contents: Newly learned formatting preferences (e.g., "keep tables to 4 columns") and a log of major architectural decisions.
📚 Authority: Mirrors the core philosophy behind Retrieval-Augmented Generation (RAG) Architecture, ensuring trusted, static context is injected into the prompt before execution to reduce hallucination.

📄 TASKS.md

Prompt Element: Current Task Queue
  • Purpose: Keeps the AI anchored during complex, multi-step operations.
  • Contents: A living, indented checklist ([ ], [/], [x]) of the active sprint objectives.
📚 Authority: Backed by Andrew Ng's recent work on "Agentic Workflows" (emphasizing iterative Planning over zero-shot outputs) and landmark research on Chain-of-Thought (CoT) Prompting (Wei et al., 2022).

📁 skills/

Prompt Element: Reusable Tasks & Output Formats
  • Purpose: Turns repetitive, multi-hour jobs into single-command instructions.
  • Contents: Sub-folders containing specific SKILL.md files (e.g., a perfect sequence of prompts to draft a blog post or deploy a server).
📚 Authority: Supported by the concept of "Least-to-Most Prompting" (Zhou et al., 2022), which proves LLMs perform exponentially better when large workflows are packaged into sequential, discrete sub-problems.

📁 Archive/

InfoSec Layer: Availability & Integrity
  • Purpose: Protects data integrity and ensures a recoverable workspace.
  • Contents: Time-stamped backups of files before the AI edits them, plus a HISTORY.md file that acts as an audit log of past chat sessions.

💬 The Active Chat Prompt

Prompt Element: One-Off Task & Output Format
  • Purpose: Your actual, typed message to the AI.
  • Contents: Because the files above handle the Identity, Context, and Constraints, your chat prompt only needs to specify the immediate micro-task and how you want the answer formatted.

The "Thin Prompt" Workflow

By offloading 80% of your prompt into the file structure above, your daily prompts become incredibly lightweight and powerful. Here is what a fully synthesized prompt looks like in practice:

"Review the code in @context/auth-module.py. Find the bug causing the login timeout (Task). Return the fixed code snippet as a markdown block with inline comments explaining the fix (Output Format)."

*The AI automatically combines this short instruction with your persistent Identity, Context, and Constraints to execute the task flawlessly.

📚 Authority: On formatting workflows in HTML/Markdown—Andrej Karpathy heavily champions this, shifting from Markdown (for simple token efficiency) to HTML for creating structured "LLM Wikis" that allow for rich semantic rendering between multi-agent systems.
